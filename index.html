GraphDSL

Creator

Generates type-safe Kotlin DSL query builders straight from your .graphqls schema files. No hand-written query strings, no extra serialisation layer — drop in the Gradle plugin, put your schema in src/main/graphql/, and the builders appear at compile time. It grew out of the same instinct as the codegen work above: schema changes should propagate to consumers automatically, not by hand.

Selected impact

Things I've shipped

Eight years of backends across airline, e-commerce and infrastructure systems. Some of the results I'm most pleased with:

  1. 98%

    Of bot traffic stopped at the door

    Designed and launched a device-integrity validation service on Spring Boot, built on Google's and Apple's integrity APIs. It cut request volume from 500M to 150M a month — a 70% drop in server load — and forced whatever was left to run on real hardware.

  2. <70ms

    Query response across 1M+ customer records

    Brought a large customer database back under control with composite indexes on hot columns, selective denormalisation to cut join overhead, and a partitioning scheme that held performance steady as the dataset kept growing.

  3. Play 2 → Boot

    Led a full airline backend rewrite

    Took a legacy Play 2 codebase in Java and Scala to Spring Boot and Kotlin, restructured as multi-module microservices with modern DevOps practice. Shared service libraries enforced consistency and removed about 70% of the duplicated code along the way.

  4. <100ms

    Real-time boarding passes and booking cards

    Core airline services generating boarding passes on demand, with payment processing across multiple gateways at 99.9% uptime and Google and Apple Wallet integration that reached 80% adoption across all flights.

  5. ~30%

    Faster builds through better tooling

    Cut build times and improved CI throughput with Bazel and Gradle — smarter caching, parallelised tasks, and the removal of redundant compilation steps. A custom Gradle codegen plugin then generated client-ready Kotlin straight from GraphQL schemas, along with scaffolding for end-to-end tests.

  6. <20% → 80%

    Test coverage across 13 microservices

    Wrote 85% of the suite myself and set the testing standards that kept it there, with logging good enough to catch the runtime failures that used to reach production.

Approach

How I like to build

01

Make the compiler do the arguing

If a mistake can be caught at build time, it should never make it to a test, let alone production. That instinct is most of why GraphDSL exists.

02

Boundaries before frameworks

Clean and hexagonal architecture aren't ceremony — they're what lets a codebase survive its third rewrite of the transport layer. Domain first, plumbing second.

03

One schema beats fifty integrations

Working on Viaduct sharpened this: the cost of a distributed system is rarely the services, it's the N×N ways they talk to each other.

04

Automate the boring parts first

Codegen, test scaffolding, build caching. Every hour spent there comes back across the whole team, and it's usually the cheapest performance work available.

05

Build it in the open

Public code gets better review, better docs and a longer life. It's also the fastest way I've found to learn from people better than me.
